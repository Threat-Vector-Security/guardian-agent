# Agentic DAG Planner Specification (Phase 1)

## Overview
This specification details the implementation of Phase 1 of the Agentic Uplift Plan: The Meta-Planner DAG. The goal is to evolve the `IntentGateway` and introduce a `TaskPlanner` and `AssistantOrchestrator` to handle complex, multi-step intents using a Directed Acyclic Graph (DAG) of sub-tasks. It incorporates Hermes-inspired programmatic tool orchestration and first-class delegation.

## Core Components

1.  **Intent Gateway Updates (`src/runtime/intent-gateway.ts`)**
    *   **Classification:** Enhance the gateway to classify intents as "complex" (requiring planning).
    *   **Routing:** Route "complex" intents to the new `TaskPlanner`.

2.  **Task Planner (`src/runtime/planner/task-planner.ts`)**
    *   **Responsibility:** Given a complex objective, generate a DAG of actionable sub-tasks.
    *   **DAG Structure:** Each node represents a specific action, tool call, subagent delegation (`delegate_task`), or programmatic execution (`execute_code`).
    *   **Dependencies:** Nodes define dependencies (edges) to ensure correct execution order.

3.  **Assistant Orchestrator (`src/runtime/planner/orchestrator.ts`)**
    *   **Responsibility:** Execute the DAG generated by the `TaskPlanner`.
    *   **Execution Engine:** Traverse the DAG, executing nodes whose dependencies are met.
    *   **State Management:** Track the status of each node (pending, running, success, failed).
    *   **Hermes Uplift - Programmatic Execution:** Allow nodes to run sandboxed code (`execute_code`) that calls managed tools and shapes data, reducing LLM roundtrips.
    *   **Hermes Uplift - Delegation:** When a node delegates via `delegate_task`, the orchestrator spawns a brokered worker with specific tool bounds and trust scoping, resuming when the subagent completes.

## Data Models

```typescript
// Proposed DAG Node Structure
interface PlanNode {
  id: string;
  description: string;
  dependencies: string[]; // IDs of nodes that must complete first
  actionType: 'tool_call' | 'skill_delegation' | 'routine_execution' | 'execute_code' | 'delegate_task';
  target: string; // Tool name, skill ID, routine ID, or code language
  inputPrompt: string; // The specific instruction, subagent prompt, or code payload
  status: 'pending' | 'running' | 'success' | 'failed';
  result?: unknown;
}

interface ExecutionPlan {
  id: string;
  originalObjective: string;
  nodes: Record<string, PlanNode>;
  status: 'planning' | 'executing' | 'completed' | 'failed';
}
```

## Security & Integration Constraints
*   **No Interference:** The `TaskPlanner` is a meta-planner. It delegates logic to specialized skills or subagents.
*   **Approvals:** The `AssistantOrchestrator` must respect the existing Guardian approval layers.
*   **Memory:** The orchestrator must not bypass the Second Brain. Routine executions flow through established boundaries.
